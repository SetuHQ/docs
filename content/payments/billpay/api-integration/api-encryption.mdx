## API payload encryption (AES-CBC)

**API payload encryption is optional.** Enable it only if your integration requires encrypted JSON bodies on the COU HTTP API and/or encrypted webhook payloads. When enabled, your **integration contact (Setu)** will confirm:

- whether you use **static IV** or **dynamic IV**
- whether the encrypted blob in `data` uses **`hex`** or **`base64`** encoding

Implement exactly what they specify for your environment—do not assume dynamic IV or a particular encoding.

Some bank programs (e.g. **FX Retail**) use **Dynamic IV** only; others use **static IV**. Follow the **Dynamic IV** or **Static IV** subsection below accordingly—your contact tells you which applies.

**Scope:** When encryption is on for your integration, it applies consistently for your partner across:

- **Inbound** API requests (Partner → COU)
- **Outbound** API responses (COU → Partner)
- **Outbound** webhooks and callbacks (COU → Partner)

You cannot mix encrypted and plaintext modes per API for the same partner; behavior is uniform once enabled.

This page covers **cryptography** and the `` `{ "data": … }` `` **envelope only.** Base URLs, authentication, individual API routes, and plaintext request/response schemas are documented in your **main BBPS COU integration** materials—use those alongside this add-on.

### Cryptographic specification

| Parameter | Value |
| --- | --- |
| Encryption | Symmetric |
| Algorithm | AES |
| Mode | CBC |
| Block size | 128 bits (16 bytes) |
| Key size | 128, 192, or 256 bits (16, 24, or 32 bytes) |
| IV (static mode) | 128 bits (16 bytes), **fixed** per partner (from configuration) |
| IV (dynamic mode) | 128 bits (16 bytes), **random per message** (included in `data`) |
| Padding | PKCS5 / PKCS7 |
| `data` encoding | **`hex`** or **`base64`** (your contact confirms which) |

Plaintext before encryption is always **UTF-8 JSON**.

#### Interoperability (avoid common mistakes)

- **Key and static IV material:** Provisioning may supply the shared key (and static IV, if used) as **hex strings** or as **raw / byte-oriented** values. You must decode and use them **exactly** as your integration contact describes—wrong interpretation is the most frequent cause of “invalid request” decrypt failures.
- **`payloadEncoding` = `base64`:** Use **standard Base64** (RFC 4648 alphabet, including `+` and `/`, with padding)—**not** URL-safe Base64—unless your contact specifies otherwise.
- **`payloadEncoding` = `hex`:** COU uses **lowercase** hex digits (`a`–`f`) when encoding; accept lowercase when decoding for robustness.
- **`isHex` (partner provisioning):** This flag applies only to how the **`key`** and **`iv`** values in `webhookEncryptionConfig` are interpreted: `true` means those fields are **hex strings** (decode hex → bytes before use); `false` means they are **literal byte strings** (use UTF-8 / raw bytes as agreed). It does **not** change how the HTTP body field **`data`** is encoded—that is controlled separately by **`payloadEncoding`** (`hex` or `base64` for the ciphertext / IV+ciphertext blob).

### Static IV vs dynamic IV (how each flow works)

#### Dynamic IV

1. Build your JSON plaintext (UTF-8), PKCS5-pad to a multiple of 16 bytes.
2. Generate a **random 16-byte IV** for this message.
3. AES-CBC encrypt using your **shared key** and that IV.
4. **Concatenate** raw bytes: `IV (16 bytes) || ciphertext`.
5. Encode that byte string with your agreed **`payloadEncoding`** (`hex` or `base64`) and put the result in JSON as `\{ "data": "<encoded>" \}`.

**Decrypt:** Decode `data` → first **16 bytes** are the IV, remainder is ciphertext → AES-CBC decrypt with the same key → unpad → parse JSON.

The IV is **not** stored in partner config for encryption math—it is **only** in the envelope (except you may still have config fields for keys/flags as provisioned).

#### Static IV

1. Use the **fixed 16-byte IV** agreed with your contact (derived from provisioning—same IV for encrypt and decrypt).
2. PKCS5-pad plaintext JSON, AES-CBC encrypt with **shared key + static IV**.
3. Encode **only the ciphertext bytes** (no IV prefix) with **`hex`** or **`base64`** → `\{ "data": "<encoded>" \}`.

**Decrypt:** Decode `data` to ciphertext bytes → AES-CBC decrypt using **shared key** and the **same static IV** from provisioning → unpad → parse JSON.

### Prerequisites (who turns what on)

**Partner ↔ COU (HTTP API):** Encrypted request/response bodies on these routes need **service-level API payload encryption** (`APIEncryption`) **and** partner encryption settings to be enabled together for that path. Your contact confirms when this is active for staging/production.

**Outbound webhooks (COU → your server):** Encryption follows **partner encryption settings** for delivery (not gated by the same HTTP `APIEncryption` switch). Your contact aligns webhook behavior with what you test.

#### Partner settings (vocabulary)

Setu stores encryption flags under **`webhookEncryptionConfig`**. You rarely edit this yourself; use the table to align with what your contact says is provisioned:

| Field | Meaning |
| --- | --- |
| `isEnabled` | Partner-level encryption on (`true`) or off (`false`) for the flows your contract covers. |
| `useDynamicIV` | `true` → Dynamic IV (IV inside `data`). `false` → static IV (fixed IV from provisioning; `data` is ciphertext only). |
| `key` | Shared AES key (see [Interoperability](/payments/billpay/api-integration/api-encryption#interoperability-avoid-common-mistakes) for hex vs raw). |
| `iv` | **Dynamic IV:** not used for the crypto envelope (may be empty). **Static IV:** must be **16 bytes** when `useDynamicIV` is false. |
| `isHex` | If `true`, **`key`** and **`iv`** in config are **hex-encoded** strings (decode to bytes). If `false`, they are **raw / literal** strings. Independent of **`payloadEncoding`** (which only describes the **`data`** field). |
| `payloadEncoding` | How the `data` string encodes bytes: **`hex`** or **`base64`** (default **`hex`** if unset). |

### HTTP headers

#### Request headers

| Header | Required | Notes |
| --- | --- | --- |
| `X-PARTNER-ID` | Yes | Your partner identifier. |
| `X-ENC-KEY-ID` | Yes (contract) | Send **`v1`** unless your contact specifies otherwise. Always send it for stable behavior. |
| `X-SESSION-ID` | Only if the **API** requires it | See warning below. |

> **Warning**  
> If you send **both** `X-PARTNER-ID` and `X-SESSION-ID`, COU may treat the call as a **whitelabel / session** path and **not** decrypt the body as an encrypted envelope. For standard encrypted partner API calls, use **`X-PARTNER-ID`** and **`X-ENC-KEY-ID`** (plus any other headers that API requires).

#### Response headers

When response encryption is active, COU typically echoes:

- `X-ENC-KEY-ID: v1` (or the value you sent)

This header is reserved for future key rotation; today cryptographic material comes from your provisioned key and IV mode.

### Encrypted JSON envelope

All encrypted bodies use a single field:

<CodeBlockWithCopy language="json">{`{
  "data": "<string>"
}`}</CodeBlockWithCopy>

What goes inside `data` depends on **IV mode** (see above):

- **Dynamic IV:** encoded bytes = **IV ‖ ciphertext** (after decoding, split at byte 16).
- **Static IV:** encoded bytes = **ciphertext only** (IV comes from provisioning, not from `data`).

Example (**dynamic IV**, `payloadEncoding` = `base64`):

<CodeBlockWithCopy language="json">{`{
  "data": "Y2I2SUx2T2h4Wk1IQWlQb0RrWWxQb1Bmb0hYdWxEcE5IWVQ5R2hLSDRkcVh2NWVpS0xNQW9Kdk5YN1k="
}`}</CodeBlockWithCopy>

### Integration flows (short)

**Encrypted API request (Partner → COU):** Build JSON → encrypt per your **IV mode** and **encoding** → POST `\{ "data": "..." \}` with `Content-Type: application/json` and required headers.

**Encrypted API response (COU → Partner):** Read `data` → decode → decrypt per **IV mode** → parse JSON.

After decryption you get a normal JSON object (structure depends on the API). Illustrative example only:

<CodeBlockWithCopy language="json">{`{
  "data": {
    "refId": "D6KL9AI5HV4HRLESG8R04hLGZVt60641532"
  },
  "status": "Processing",
  "traceId": "D6KL9AI5HV4HRLESG8QG"
}`}</CodeBlockWithCopy>

**Encrypted webhooks:** Same `\{ "data": "..." \}` envelope; decrypt the same way as API responses for your mode. Expose an **HTTPS** URL for webhooks; retry behavior and delivery semantics follow your onboarding or SLA with Setu.

### Error handling

If COU cannot decrypt your request, you may receive a plaintext error such as:

<CodeBlockWithCopy language="json">{`{
  "error": {
    "code": "invalid-request",
    "message": "Invalid request"
  },
  "success": false,
  "traceId": "<trace-id>"
}`}</CodeBlockWithCopy>

**Common causes:**

- Wrong or rotated key not yet reflected in the environment you are calling  
- Wrong **IV mode** (treating static as dynamic or vice versa)  
- Wrong **hex vs base64** on `data`, or **standard Base64** vs URL-safe mix-ups  
- Envelope missing `data`, or **dynamic IV** payload too short (must contain at least 16-byte IV + ciphertext after decode)  
- `X-PARTNER-ID` + `X-SESSION-ID` together when encryption was expected  
- Partner settings updated but **cache** not yet showing the new config (retry after a short wait or contact Setu)

**Support:** When escalating to Setu, include the **`traceId`** from the error response and any request correlation id you use.

### Checklist

1. Confirm **API payload encryption** (`APIEncryption`) is on for the **COU HTTP API** environment used for Partner ↔ COU calls (if you use encrypted API calls). Not used as a gate for outbound webhook delivery.
2. Confirm partner **`webhookEncryptionConfig`** matches what you built: **`isEnabled`**, **`useDynamicIV`** (true for Dynamic IV flows), **`payloadEncoding`**, and **key/IV encoding** (`isHex`).
3. Confirm with your contact: **static IV vs dynamic IV**, **`hex` vs `base64`** for `data`, and **how key/IV strings are encoded** (hex vs raw bytes).
4. Exchange and store the shared key securely (and static IV material if applicable).
5. Send requests using the encrypted envelope with **`X-PARTNER-ID`** and **`X-ENC-KEY-ID`**. Body: `\{ "data": "…" \}` (**dynamic IV:** encoded IV+ciphertext; **static IV:** encoded ciphertext only).
6. Do **not** send `X-SESSION-ID` on encrypted partner calls unless the API explicitly requires it.
7. Decrypt responses and webhooks by decoding `data`, then decrypting per your mode (**dynamic IV:** split IV, then decrypt; **static IV:** decrypt with provisioned IV).
8. Test against **staging / sandbox** before production cutover.
